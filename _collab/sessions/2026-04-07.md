### Feature 1: D3 Force Graph in /explore (`src/app/explore/page.tsx`)

**Status:** ✅ Complete — full rewrite

Replaced react-cytoscapejs (Cytoscape) with a pure D3 canvas-based force-directed graph. The old Cytoscape implementation rendered as a black canvas with no visible nodes on most environments.

**What was built:**
- Full D3 force simulation using `d3.forceSimulation` with link, charge, center, and collision forces
- Canvas rendering with `requestAnimationFrame` draw loop
- Node colors by entity_type: company=#0EA5E9, technology=#10B981, industry=#F59E0B, product=#8B5CF6, problem=#EF4444, signal=#6B7280
- Node size proportional to degree (relationship count), min 4px, max 14px
- Drag-to-move nodes (D3 fx/fy pinning) + canvas pan/scroll zoom (wheel event)
- Click node → right panel shows entity name, description, connected entities + relationship types
- Filter chips: All | Company | Technology | Industry | Product
- Search bar to highlight a matching node in sky-blue
- "Zoom to fit" button that calculates bounding box and fits all nodes
- Stats header: "{N} entities · {M} relationships"
- ResizeObserver to handle container size changes
- Legend at bottom-left showing type color key
- AnimatePresence for panel transition
- Dark background #07090A throughout

**Technical notes:**
- Uses `useRef` for simulation, transform, nodes, links, animFrame — no re-renders during simulation
- Transform state tracked via `d3.ZoomTransform` ref to avoid React re-render on every frame
- Highlight node ID tracked via ref (not state) to avoid triggering re-renders
- Type fix: `ctx.moveTo(s.x ?? 0, s.y ?? 0)` to handle D3 `number | undefined` node coords

### Feature 2: Ask Jarvis — Real Conversational Intelligence (`src/app/api/ask/route.ts`)

**Status:** ✅ Complete — full rewrite

Replaced the complex multi-engine ask pipeline (intent detection, 5 Whys, signal connections, decision cards) with a focused Jarvis-style strategic intelligence endpoint that queries live Supabase data and synthesizes it with Gemini.

**What was built:**
- Queries `intel_signals` (filtered by query, sorted by importance_score, excludes arxiv)
- Queries `vendors` (filtered by query, returns top 5)
- Queries `kg_discoveries` (filtered by query, returns top 5)
- All 3 queries run in `Promise.allSettled` for resilience
- Sends data to Gemini via `runParallelJsonEnsemble` with `preferredProviders: ['gemini']`
- Jarvis system prompt structures answer into 5 sections: THE SITUATION, KEY PLAYERS, WHY EL PASO CARES, WHAT TO WATCH, RECOMMENDED ACTION
- Graceful fallback `buildFallbackAnswer()` if LLM unavailable
- Returns: `{ ok, answer, signals, vendors, discoveries, query, provider, generated_at }`

**Design decision:** The old route was architecturally more complex (assembleAskResponse → intent detection → 5 Whys → connections → decision card) but added latency and complexity. The new Jarvis route is direct: Supabase query → LLM synthesis → structured answer. Much easier for the frontend to render.

### Feature 3: Patent Pipeline (`src/app/api/agents/patent-discovery/route.ts`)

**Status:** ✅ Complete — new file

Builds a recurring agent that fetches real patents from the PatentsView / USPTO public API and persists them to `kg_discoveries` as `discovery_type: 'patent'`.

**What was built:**
- POST handler with optional CRON_SECRET auth
- 6 sector queries: AI/ML, Logistics, Defense, Border Tech, Manufacturing, Energy
- Each query calls `https://api.patentsview.org/patents/query` (no API key required)
- Fetches 20 patents per sector, sorted by patent_date desc
- Deduplication by title against existing `kg_discoveries` rows
- IKER score calculation:
  - Base: 50
  - +20 if assignee is known high-value org (Lockheed, Google, MIT, DARPA, etc.)
  - +10 if CPC section G or H (Physics / Electricity)
  - +15 if 2024+, +8 if 2023+
- Inserts into `kg_discoveries` with: title, summary (abstract), discovery_type='patent', source_url (Google Patents), source_name='PatentsView / USPTO', research_institution (assignee), published_at, iker_impact_score
- Returns full stats: total_inserted, total_skipped, total_errors, sectors_processed, per-sector breakdown
- GET handler returns agent metadata for health checks
- vercel.json updated with cron: `0 10 * * *` (daily 10:00 UTC)
